# D-0005: 結構驗證嚴格度 — 自連禁、孤兒警告、環狀放行

- **Status**: Accepted
- **Date**: 2026-05-14
- **Affects**: §3.4 (BR-4), §4.2 (EF-3), §4.3 (EC-3, EC-4, EC-5, EC-6)
- **Supersedes**: —
- **Superseded by**: —

## Context

匯入模板時可能遇到各種「結構上不尋常」的情況：
- 節點自連（A → A）
- 孤兒節點（未連線的節點）
- 同兩節點間多條連線
- 環狀連線（A → B → C → A）
- 單一節點模板

需要決定每種情況是「擋下不允許匯入」、「允許但警告」、還是「完全放行」。

過嚴會擋住合理使用情境；過鬆會放任明顯錯誤進入系統。

## Decision

按以下嚴格度分類：

| 結構 | 處理 |
|------|------|
| 節點自連（source == target） | **禁止**，匯入驗證失敗 |
| 孤兒節點（節點數 > 1 且存在未連線節點） | **警告**，允許匯入但預覽頁提示 |
| 同兩節點間多條連線 | **警告**，允許匯入但預覽頁提示 |
| 環狀連線（A → B → C → A） | **完全放行**，不警告 |
| 單一節點模板（無 connection） | **完全放行**，不警告 |

## Rationale

**禁止節點自連**：技術上沒有合理用途（同一節點觸發自己會死循環或永遠不觸發），是明顯錯誤。

**警告但允許孤兒節點**：使用者可能正在編輯中，孤兒節點是「未完成」狀態。匯入後在編輯器中補連線是合理流程。擋下會強迫使用者在匯入前手動修正，反而增加 friction。

**警告但允許多重連線**：可能是刻意設計（不同條件分別觸發），也可能是錯誤。系統不應替使用者判斷，但提示「建議改用 OR 條件合併」。

**完全放行環狀連線**：業務上常見合理場景（例：審核退回重審、循環審批）。擋下會傷害實際需求。

**完全放行單一節點**：是合法的初始狀態，新建模板第一步就是單一節點。

## Alternatives Considered

### A1: 全部允許，只在執行時偵測問題

**Pros**:
- 匯入流程最簡單
- 不需要 validation logic

**Cons**:
- 明顯錯誤（自連）進到系統會造成執行時 bug
- 使用者沒有提前 feedback

**Why rejected**: 自連這種明顯錯誤應在匯入時擋下，避免污染資料

### A2: 嚴格擋下所有「不尋常」結構（含環狀、孤兒）

**Pros**:
- 確保進入系統的模板都「結構乾淨」

**Cons**:
- 擋住合理的環狀場景（審核退回）
- 強迫使用者修正所有警告才能匯入，增加 friction
- 對「先匯入再編輯」的使用流程不友善

**Why rejected**: 環狀有正當用途；孤兒在編輯中是正常狀態

### A3: 只擋自連，其他全放行（不警告）

**Pros**:
- 比 A2 寬鬆，不擋合理場景

**Cons**:
- 失去「告知使用者潛在問題」的機會
- 孤兒節點和多重連線可能是錯誤，使用者沒收到提示就匯入了

**Why rejected**: 警告對使用者有資訊價值，且不擋匯入也不算 friction

## Consequences

**Positive**:
- 平衡了「擋掉錯誤」跟「不過度限制」
- 警告制讓使用者保有決策權
- 環狀放行支援審核退回等真實場景

**Negative**:
- 預覽頁需要設計警告區塊（C-6 額外複雜度）
- 警告規則未來可能要擴充（例：未來新增「連線數量過多」等規則時要重新討論）
- 「警告但允許」對 QA 不易測（要驗證警告顯示對 + 匯入成功對）
